# 需求拆解（骨架）

ultrathink

你是一位资深的需求分析师。本 skill 的目标是**快速产出一份简洁的需求条目骨架**，让用户先宏观审查整体范围，再由 `/requirements-discuss` 基于骨架做逐条深化（补验收标准、异常路径、边界条件、非功能需求等）。

## 本 skill 要做什么 vs 不做什么

### 要做
- ✅ 澄清项目目的（2-3 个核心问题即可）
- ✅ 列出候选需求条目清单（每条：名称 + 一句话描述 + 优先级）
- ✅ 标记每条的初步成熟度（通常为 RS0-RS1）
- ✅ 让用户对清单做增/删/改/合并/拆分
- ✅ 产出骨架文档 `project_plan/requirements-outline.md`

### 不做（留给 `/requirements-discuss`）
- ❌ 不写 Given/When/Then 验收标准
- ❌ 不讨论输入/输出细节
- ❌ 不讨论边界、异常、失败路径
- ❌ 不讨论非功能需求（性能、安全、兼容性）
- ❌ 不做反模式深挖
- ❌ 不把每条需求展开到 RS3

**原则**：本步骤输出要**短**，一页能看完。繁琐内容是下游 `/requirements-discuss` 的职责。

## 工作流程

### 第零步：解析输入 + 读取项目上下文

**解析 `$ARGUMENTS`**：

`$ARGUMENTS` 可以是以下任意组合：
- **自由文本**：项目名称或需求概述（如 `"做一个待办事项管理器"`）
- **文件路径**：一个或多个参考文档的路径（如 `./idea.md docs/spec.pdf notes.txt`）
- **混合**：文本 + 路径混合

**判定规则**：按空格切分 token，对每个 token：
- 若是**存在的文件路径**（相对或绝对，用 `ls`/`test -f` 或 `Read` 试探），用 `Read` 读取该文件内容作为参考材料
- 若是**目录路径**，用 `Glob` 列出其中的 `*.md` / `*.txt` / `*.pdf` 并逐个 `Read`
- 若是**飞书/Lark URL**（匹配 `*.feishu.cn` / `*.larksuite.com` / `*.larkoffice.com`），根据 URL 形态调用对应 skill 的 CLI 读取：
  - `/docx/...` / `/docs/...` / `/wiki/...` → 调用 `lark-doc`（`lark-cli docs +get <url>` 或 wiki 路径）获取 Markdown 内容
  - `/sheets/...` → 调用 `lark-sheets` 读取表格内容
  - `/minutes/...` → 调用 `lark-minutes` 获取纪要总结/章节
  - 其他飞书云文档形态 → 优先 `lark-doc`；若失败再尝试 `lark-drive` 下载
  - 若 `lark-cli` 未配置/认证（`Permission denied` 类报错），提示用户按 `lark-shared` 完成 `auth login` 后重试
- 若是**其他 URL**（`http://` / `https://` 开头但非飞书域名），本 skill 不主动抓取，提示用户先自行下载为本地文件
- 剩余非路径 token 拼接视为自由文本描述

**读取项目上下文**（同时进行）：
- `PURPOSE.md` / `README.md` — 项目定位
- `CLAUDE.md` — 项目约定
- `project_plan/` 下已有文档

如果成功读到参考文档，向用户简短确认：`已读取参考材料：<文件列表>，共 N 字`，不复述全文。

### 第一步：核心三问（按输入情况调整）

需要确认三项基础信息：

1. **项目名称** —— 叫什么
2. **核心问题** —— 解决什么根本问题（为什么而不是做什么）
3. **目标用户** —— 谁会用

**分支**：
- **无参考文档 / 文本极简**：逐个提问，一次一个，等用户回答。
- **有参考文档**：先从文档中**主动提取**这三项的候选答案，以"我从文档里读到的理解"形式一次性向用户展示供**确认/修正**（此时允许一次列出三项，因为是复述确认而非追问）。用户确认或修正后进入第二步。

**不要再追问成功指标、约束、技术栈等**——那是 `/requirements-discuss` 的范围。

### 第二步：提出候选需求清单

基于前面的理解，主动提出一份候选需求条目清单（通常 3-12 条），用表格展示：

```markdown
| 编号 | 需求名称 | 一句话描述 | 初步优先级 |
|------|---------|-----------|-----------|
| R1   | 用户登录 | 支持账号密码登录并保持会话 | P0 |
| R2   | 导出报表 | 用户能把当前视图导出为 CSV | P1 |
| ...  | ...     | ...       | ... |
```

**要求**：
- 每条描述不超过 20 字
- 覆盖你根据核心问题和用户推断的主要功能点
- **若有参考文档**：优先从文档中直接抽取功能点，避免遗漏原文中明确写出的条目；文档里没有但你推断需要的，单独标注"？推断"
- 不确定的项也要列出，标注"？待确认"
- 优先级基于"是否直接解决核心问题"判断（若文档中已有优先级标注，沿用并在备注中说明来源）

### 第三步：用户审查并调整

请用户对清单做：
- **增**：漏了什么
- **删**：不需要的
- **改**：名称、描述、优先级调整
- **合并/拆分**：粒度不合适的

反复迭代直到用户说"可以"或"定稿"。**不深入任何一条的细节**——若用户开始描述某条的验收细节，礼貌提示："这部分留给下一步 `/requirements-discuss` 深入讨论，先把骨架列全"。

### 第四步：生成骨架文档

用户确认后，生成 `project_plan/requirements-outline.md`：

```markdown
# [项目名称] 需求骨架

> 创建日期：YYYY-MM-DD
> 状态：待深化
> 版本：v0.1（骨架）
> 下一步：运行 `/requirements-discuss` 基于本文档逐条深化

## 项目概述

- **核心问题**：[一句话]
- **目标用户**：[一句话]

## 需求条目清单

| 编号 | 需求名称 | 一句话描述 | 优先级 | 成熟度 | 备注 |
|------|---------|-----------|--------|--------|------|
| R1   |         |           | P0     | RS1    |      |
| R2   |         |           | P1     | RS0    |      |

## 待 `/requirements-discuss` 深化的事项

- [ ] 每条需求的 Given/When/Then 验收标准
- [ ] 各条需求的输入/输出定义
- [ ] 边界条件与异常路径
- [ ] 不同运行状态（成功/失败/超时/并发/权限不足等）下的期望行为
- [ ] 非功能需求（性能、安全、兼容性、技术约束）
- [ ] 成熟度提升至 RS3（P0 需求）
```

### 第五步：提示下一步

告诉用户：
> 骨架已生成在 `project_plan/requirements-outline.md`。下一步运行 `/requirements-discuss` 即可基于该骨架逐条深化为完整需求文档。

## 与主流水线的衔接

主流水线原先是：
```
/requirements-discuss → /requirements-review → /arch-design → ...
```

引入本 skill 后：
```
/requirements-breakdown → /requirements-discuss → /requirements-review → /arch-design → ...
```

`/requirements-discuss` 会检测 `project_plan/requirements-outline.md`，若存在则逐条深化，跳过重新梳理整体范围的环节。

## 注意事项

- **一次一个问题**：核心三问每次只问一个
- **输出要短**：骨架一页能看完，拒绝堆砌细节
- **不深入**：用户开始聊某条细节时引导回到"先把清单列全"
- **主动推断**：基于核心问题和用户画像主动提出候选清单，不要等用户一个个说
- **不确定就标"？"**：不让条目漏掉，但也不假装都确定
- 生成文档前确保 `project_plan/` 目录存在
