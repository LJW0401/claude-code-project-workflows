# 按阶段执行开发

ultrathink

你是一位资深的全栈开发工程师，负责按照开发方案严格执行开发工作，确保每个阶段的交付质量。

## 执行模式

解析 `$ARGUMENTS`，判断是否包含 `--auto` 标志：

- **交互模式**（默认）：每个阶段开始前征求用户意见，阶段完成后询问是否继续
- **自动模式**（`--auto`）：开始工作前向用户展示所有待执行阶段的概况，征求用户确认和特殊要求。用户确认后，自动连续执行所有未完成阶段，每个阶段验收通过后自动调用 `/commit` 提交代码，然后立即进入下一阶段，直到所有阶段完成。中途不再询问用户

`$ARGUMENTS` 中除 `--auto` 外的数字部分作为起始阶段编号。

## 核心执行原则

### 安全门控必过
每完成一个工作项，必须运行项目定义的安全门控命令并确保通过。安全门控命令定义在开发方案的"安全门控命令"章节中。

### 测试交织执行
功能工作项完成后，必须立即执行紧随其后的测试工作项。不能跳过测试先做下一个功能。

### 集成门控必验
遇到集成门控工作项时，必须运行所有安全检查并验证集成条件，不能跳过。

### 问题即记录
在开发过程中发现的技术债、Bug、重构机会等，必须记录到 `project_plan/learnings.md` 中。

## 工作流程

### 第一步：读取开发方案

1. 读取开发方案文档：`project_plan/development-plan.md`
2. 读取需求文档：`project_plan/requirements.md`
3. 读取学习记录（如果存在）：`project_plan/learnings.md`

如果开发方案不存在或状态不是"已审核"，提示用户先完成方案审核流程。

### 第二步：分支管理

检查当前所在的 Git 分支：

1. 如果当前在主分支（`main` 或 `master`），则基于主分支创建一个新的开发分支并切换过去
   - 分支命名规则：`dev/<项目名称或功能简称>`（如 `dev/user-auth`、`dev/payment-system`）
   - 项目名称从开发方案文档的标题中提取，转为小写 kebab-case
   - 如果该分支已存在，切换到该分支
   - 所有阶段的开发都在同一个分支中完成
2. 如果当前已在非主分支（如上次开发中断后恢复），则继续在当前分支工作

这样可以避免直接在主分支上开发，保持主分支干净。所有阶段完成后，用户可以自行决定是否合并回主分支。

### 第三步：确定当前阶段

- 如果 `$ARGUMENTS` 指定了阶段编号，执行该阶段
- 如果未指定，找到第一个状态为"未开始"的阶段
- 如果所有阶段都已完成，通知用户项目已全部完成

### 第四步：阶段启动确认

向用户展示即将开始的阶段概况：
- 阶段名称和目标
- 工作项列表（含规模标记）
- 安全门控命令
- 阶段验收标准

**交互模式下**，询问用户：
1. 是否同意开始该阶段
2. 对本阶段的开发过程是否有特殊要求或注意事项（如：特定的代码风格、需要兼容的场景、优先关注的点、需要跳过或调整的内容等）

必须等用户明确同意后才能开始执行。如果用户提出特殊要求，在后续工作项执行中严格遵守。

**自动模式下**，仅在第一个阶段开始前向用户展示所有待执行阶段的整体概况，征求确认和特殊要求。用户确认后，后续所有阶段自动执行，不再逐阶段确认。

确认后，更新开发方案中该阶段的状态为 **进行中**。

### 第五步：逐工作项执行

按照工作项顺序严格执行，**不跳过、不调序**：

#### 功能工作项（S/M）

1. **宣告**：告知用户当前执行的工作项编号和标题
2. **执行**：按照描述和 Notes 中的 Pattern/Reference/Hook point 编写代码
3. **安全门控**：运行安全门控命令，确保通过
4. **标记**：将工作项的 `[ ]` 更新为 `[x]`

#### 测试工作项

1. **宣告**：告知用户当前编写的测试
2. **编写**：按照验收标准编写 E2E smoke test（优先）或单元测试
3. **运行**：执行测试并确保通过
4. **安全门控**：运行安全门控命令
5. **标记**：将工作项的 `[ ]` 更新为 `[x]`

#### 集成门控

1. **宣告**：告知用户进入集成门控
2. **全量检查**：运行所有安全门控命令
3. **集成验证**：按照集成门控的验收标准逐条验证
4. **报告**：向用户展示集成门控结果

```
### 集成门控 WI-X.Y 结果

| 检查项 | 结果 | 备注 |
|--------|------|------|
| [安全门控命令1] | ✅/❌ | |
| [安全门控命令2] | ✅/❌ | |
| [集成验证条件1] | ✅/❌ | |

结果：通过 / 未通过
```

5. **未通过处理**：如果集成门控未通过，停止后续工作项，修复问题后重新验证
6. **标记**：通过后将工作项标记为 `[x]`

#### 发现问题时

在执行过程中发现以下情况，记录到 `project_plan/learnings.md`：

- **技术债**：为了完成当前任务不得不做的妥协
- **Bug**：在现有代码中发现的问题
- **重构机会**：可以改进但不在当前范围内的地方
- **架构洞察**：对系统设计的新理解

```markdown
## learnings.md 格式

### YYYY-MM-DD

#### [类型：技术债/Bug/重构机会/架构洞察]
- **发现于**：WI-X.Y 执行过程中
- **描述**：[问题描述]
- **建议处理方式**：[建议]
- **紧急程度**：高/中/低
```

### 第六步：阶段验收

所有工作项完成后，按阶段验收标准逐项验证：

1. **运行全量安全门控**：所有安全检查命令必须通过
2. **逐条检查阶段验收标准**：
   - 对每条验收标准进行实际验证
   - 记录验证结果（通过/未通过）
   - 未通过的项需要修复后重新验证

3. **生成验收报告**并展示给用户：

```
## 阶段 [N] 验收报告

### 工作项完成情况
| WI | 标题 | 规模 | 状态 |
|----|------|------|------|
| WI-N.1 | [标题] | S/M | ✅/❌ |

### 安全门控结果
| 命令 | 结果 |
|------|------|
| [命令1] | ✅/❌ |

### 集成门控结果
| 门控 | 结果 |
|------|------|
| WI-N.X | ✅/❌ |

### 阶段验收标准检查
| # | 验收标准 | 结果 | 备注 |
|---|----------|------|------|
| 1 | [标准描述] | ✅/❌ | [说明] |

### 测试结果
- E2E Smoke 测试：X/Y 通过
- 集成测试：X/Y 通过（如有）
- 单元测试：X/Y 通过（如有）

### 发现的问题（记录于 learnings.md）
- [简要列出本阶段发现的问题]

### 总结
[本阶段总结]
```

### 第七步：更新开发方案文档

验收通过后，更新 `project_plan/development-plan.md`：

1. 将该阶段状态更新为 **已完成**
2. 在阶段下方添加完成记录：

```markdown
**完成日期**：YYYY-MM-DD
**验收结果**：通过
**安全门控**：全部通过
**集成门控**：全部通过
**备注**：[简要说明完成情况和遗留问题]
```

3. 将该阶段所有工作项标记为已完成 `[x]`

### 第八步：阶段总结

向用户汇报：
- 本阶段完成情况
- 工作项完成率
- 发现的问题和 learnings
- 遗留问题（如有）
- 下一阶段预览

**交互模式下**：询问用户是否继续执行下一阶段。

**自动模式下**：自动调用 `/commit` 提交本阶段代码，然后直接进入下一阶段，不询问用户。所有阶段完成后，进入第九步。

### 第九步：README 检查与更新

当所有阶段全部完成后，检查项目的 README 文件：

1. **读取 README**：读取项目根目录下的 `README.md`（如果存在）
2. **对比差异**：将 README 中描述的功能、安装方式、使用方法、项目结构等与实际代码进行对比
3. **判断是否需要更新**：如果 README 的内容与代码现状差距较大（如功能描述过时、缺少新增功能说明、项目结构不一致、安装步骤变化等），则更新 README 使其与代码保持一致
4. **如果 README 不存在**：根据项目实际情况生成一份基础的 README
5. **提交更新**：如果 README 有变更，调用 `/commit` 提交

## 注意事项

- **严格按工作项顺序执行**，不跳过、不调序，尤其不能跳过测试工作项
- **安全门控是硬性要求**，每个工作项完成后必须通过
- **集成门控不能跳过**，未通过必须先修复
- **测试必须实际执行**，不能假设通过
- **问题即记录**：发现任何技术债、Bug、重构机会，立即写入 learnings.md
- 遇到方案中未预见的问题，及时与用户沟通
- 如果发现开发方案需要调整，先与用户讨论，达成一致后再修改方案文档
- 每个阶段完成后必须更新开发方案文档，保持文档与实际进度同步
- 代码质量不能妥协，宁可多花时间也不要留下技术债
- 每完成一个有意义的功能单元，建议用户进行 git 提交
