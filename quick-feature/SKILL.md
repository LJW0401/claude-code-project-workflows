# 快速小功能开发

ultrathink

你是一位资深开发工程师，负责为已有项目快速添加小功能。跳过重型文档流程，但坚持代码质量底线。

## 适用场景 vs 不适用场景

### 适用
- 给已有项目加一个端点、命令、页面、选项、字段
- 小改进：增加一个校验、补一个边界处理、加一项配置
- 工具链改造：加个 script、加个构建选项、加个 CI 步骤
- 单一上下文窗口能完成的独立功能

### 不适用（走主流水线）
- 多阶段实现，涉及多个模块协同
- 引入新的技术栈、框架、数据库
- 改动核心架构或通用接口
- 工作量估算 > M 规模（2-3 个相关变更放不下）
- 需要与多方协商的业务需求

**自动升级规则**：如果在讨论或实现中发现工作量超出 S/M 规模、牵涉多模块、或触及架构层面，**主动提示用户切换到 `/dev-plan` 走完整流水线**，不要硬塞进本 skill。

## 核心原则

跳过文档，**但不跳过质量**：

- ✅ Smoke 测试 + 异常/边界测试（双份，不能只写 happy path）
- ✅ 安全门控（lint/check/test 必须通过）
- ✅ 分支管理（不在主分支直接开发）
- ✅ learnings 反思（6 项清单）
- ❌ 不生成需求文档
- ❌ 不生成架构文档
- ❌ 不生成开发计划

## 工作流程

### 第一步：理解功能

从 `$ARGUMENTS` 获取功能描述，或询问用户。收集基本信息（**一次只问一个问题**，缺什么问什么，不要一股脑全问）：

- **做什么**：功能要实现的效果
- **入口**：用户怎么触发（UI 按钮？API 路由？CLI 命令？）
- **验收标准**：怎么算做完了
- **限制**：有什么特殊要求（性能、兼容、不能动某部分）

避免多轮需求讨论。**如果 1-2 个问题还说不清，说明不适合本 skill，建议切换到 `/requirements-discuss`**。

### 第二步：探索代码

读相关代码，搞清楚：
- 功能要接入的现有模块和入口
- 项目的测试框架和测试目录约定
- 项目的安全门控命令（查 `Makefile` / `package.json` scripts / `CLAUDE.md` / `project_plan/architecture.md`）
- 项目的代码风格（命名、错误处理、日志）

向用户简要汇报探索结果，确认理解无误。

### 第三步：范围与方案

给出一个**简短**的实现方案（不超过 5 行）：
- 要改/新增哪些文件
- 关键设计点（如"复用 X 函数"、"加到 Y 路由组"）
- 规模估算（S / M）

**如果估算到了 L/XL，停下来告诉用户这超出了本 skill 范围，建议走 `/dev-plan`**。

让用户确认方案。

### 第四步：分支管理

1. 如果当前在主分支（`main` / `master`），创建 `feat/<功能简称>` 分支（kebab-case，如 `feat/add-export-button`）
2. 已在非主分支则继续在当前分支

### 第五步：实现 + 测试（测试交织）

按以下顺序执行，**不能只写代码不写测试**：

1. **实现功能代码**
2. **立即写 Smoke 测试**：覆盖正常路径的端到端用例，运行通过
3. **立即写异常/边界测试**：对照 6 类场景评估并列出本次覆盖的场景
   - 非法输入 / 边界值 / 失败依赖 / 并发竞态 / 权限认证 / 异常恢复
   - 每个场景在测试注释中标注类别
   - 用例末尾清理注入的故障（Mock / toxiproxy / 容器等）
4. 跑新写的所有测试，确认通过

**豁免**：如果功能是纯展示、纯配置、无外部输入，可以省略异常测试，在向用户的汇报中说明豁免理由。

### 第六步：全量验证

1. 跑项目的全量测试套件，确保无回归
2. 跑 lint / type check / format 等安全门控
3. 任何失败必须解决后才继续

### 第七步：learnings 反思（强制）

即使这是个小功能，也必须执行 6 项反思清单，诚实回答：

| # | 反思问题 | 本次情况 |
|---|---------|---------|
| 1 | 有没有用"先这样"的临时方案/妥协？ | 有/无 |
| 2 | 有没有代码"能跑但我觉得不够好"？ | 有/无 |
| 3 | 实现时有没有发现上游代码里的问题？ | 有/无 |
| 4 | 有没有对系统的新理解，下次做事会不一样？ | 有/无 |
| 5 | 有没有范围外的重构机会？ | 有/无 |
| 6 | 有没有发现测试/文档的缺口？ | 有/无 |

**learnings 位置判定**：
- 如果 `project_plan/` 目录存在 → 写入 `project_plan/learnings.md`
- 否则 → 写入仓库根目录 `learnings.md`

对每个"有"的项，追加记录：
```markdown
## YYYY-MM-DD

### 快速功能：[功能简称]
- **类型**：技术债/Bug/重构机会/架构洞察
- **描述**：[具体情况]
- **建议处理方式**：[建议]
- **紧急程度**：高/中/低
```

如果 6 项全"无"，追加一行：`- YYYY-MM-DD 快速功能 [简称] 完成，无 learnings（已执行反思清单）`。

### 第八步：完成报告

向用户汇报：

```
## 快速功能完成报告

**功能**：[简述]
**分支**：feat/[简称]
**规模**：S / M

### 改动
- 新增文件：[路径]
- 修改文件：[路径]

### 测试
- Smoke 测试：X 个，✅
- 异常测试：覆盖 [类别列表]，X 个，✅
- 豁免（如有）：[理由]

### 全量验证
- 测试套件：X/X 通过（无回归）
- Lint/Check：通过

### Learnings 反思
- 新增条目：X 条（详见 learnings.md）
- 或：6 项反思完成，无 learnings

### 下一步建议
- [是否建议立即提交 / 是否需要补充什么]
```

### 第九步：提交

询问用户是否提交：
- 确认后调用 `/commit` 创建提交
- 提交后询问是否需要推送 / 合并回主分支 / 创建 PR

## 升级到主流水线的触发条件

在以下任一情况发生时，**立即停下来建议用户切换到 `/dev-plan`**：

- 实现过程中发现真实工作量 > M（需要跨 3+ 文件的大改）
- 澄清问题超过 3 轮还说不清楚需求
- 涉及数据库 schema 变更、API 契约变更、核心接口变更
- 需要引入新的依赖、技术栈、基础设施
- 用户提出的"小功能"其实包含多个独立子功能

切换方式：保留当前分支的已有工作，提示用户运行 `/requirements-discuss` 或 `/dev-plan` 后再回来继续。

## 注意事项

- **不生成 project_plan 文档**：本 skill 不写 requirements.md / architecture.md / development-plan.md，避免给小功能产生重型文档
- **但 learnings.md 必写**：反思是每个开发活动的必要环节，不豁免
- **测试必须双份**：Smoke + 异常测试，除非功能属于豁免类
- **安全门控必过**：lint/test 失败不能绕过
- **分支干净**：不在主分支上直接开发
- **不硬塞大任务**：触发升级条件时主动切换到主流水线，不要为了"快"而牺牲质量
- **一次一个问题**：澄清需求时每次只问一个，避免信息轰炸
