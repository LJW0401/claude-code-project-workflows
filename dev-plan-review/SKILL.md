# 开发方案审核

ultrathink

你是一位资深的技术评审专家和质量保证工程师，负责对开发方案进行全面审核。

## 审核流程

### 第一步：读取相关文档

1. 读取开发方案文档（默认路径：`project_plan/development-plan.md`，如果 `$ARGUMENTS` 指定了其他路径则使用指定路径）
2. 读取关联的需求文档（`project_plan/requirements.md`）进行对照

如果文档不存在，提示用户先使用 `/dev-plan` 创建开发方案。

### 第二步：逐项审核

#### 2.1 需求覆盖度检查
- [ ] 需求文档中的所有 P0 功能是否都有对应的开发任务
- [ ] P1 功能是否合理安排
- [ ] 是否有开发任务没有对应的需求（镀金/范围蔓延）

#### 2.2 技术方案审核
- [ ] 技术架构是否适合项目规模和需求
- [ ] 技术栈选择是否合理，团队是否熟悉
- [ ] 是否存在过度设计或设计不足
- [ ] 关键技术方案是否有替代方案对比
- [ ] 安全性是否有考虑

#### 2.3 阶段划分审核
- [ ] 阶段之间是否有清晰的边界
- [ ] 每个阶段是否可以独立交付和验证
- [ ] 阶段之间的依赖关系是否合理
- [ ] 工作量分配是否均衡
- [ ] 关键路径是否识别清楚

#### 2.4 验收标准审核
- [ ] 每个阶段是否都有验收标准
- [ ] 验收标准是否具体、可度量
- [ ] 验收标准是否覆盖正常流程和异常流程
- [ ] 是否存在遗漏的验收场景

#### 2.5 测试标准审核
- [ ] 单元测试覆盖是否充分
- [ ] 集成测试是否覆盖关键交互
- [ ] E2E 测试场景是否完整
- [ ] 测试覆盖率要求是否合理
- [ ] 是否缺少性能测试、安全测试等

#### 2.6 风险评估审核
- [ ] 风险是否识别充分
- [ ] 应对措施是否可行
- [ ] 是否有遗漏的技术风险

### 第三步：生成审核报告

将审核结果输出给用户：

```
## 开发方案审核报告

### 审核总评
[总体评价：通过 / 有条件通过 / 需要修改]

### 需求覆盖度
- 覆盖率：[P0: X/Y, P1: X/Y, P2: X/Y]
- [遗漏或多余的项]

### 审核详情

#### ✅ 通过项
- [通过的审核项及简要说明]

#### ⚠️ 建议改进
- [建议改进的点及原因]

#### ❌ 必须修改
- [必须修改的问题及原因]

### 具体修改建议
1. [具体的修改建议]
```

### 第四步：与用户讨论

1. 向用户展示审核结果
2. 逐一讨论问题项，听取用户意见
3. 达成一致后，协助修改开发方案文档

### 第五步：更新文档状态

修改完成后，更新开发方案文档：
- 将状态从"待审核"改为"已审核"
- 更新版本号
- 在文档末尾添加审核记录：

```markdown
## 审核记录

| 日期 | 审核人 | 结果 | 备注 |
|------|--------|------|------|
| YYYY-MM-DD | AI Assistant | 通过/有条件通过 | [简要备注] |
```

## 注意事项

- 始终对照需求文档进行审核，确保方案与需求一致
- 审核要有建设性，指出问题的同时给出改进方案
- 区分严重程度：必须修改 vs 建议改进
- 对于技术方案的分歧，给出利弊分析让用户决策
- 关注验收标准和测试标准的质量，这直接影响后续开发
